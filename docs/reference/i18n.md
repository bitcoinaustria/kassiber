# Localization (i18n)

Kassiber's **desktop UI** is localized with
[i18next](https://www.i18next.com/) + react-i18next. English and German ship
today; the system is built so adding a language or a string is a small,
mechanical change.

The **CLI and daemon are not localized.** They speak stable machine codes
(`kind`, error `code`, quarantine `reason`, sync `phase`) and the `--machine`
envelope is a deterministic API contract. Human-facing prose is the GUI's job:
the UI maps codes to translated copy. Do not push locale awareness into the
daemon or change `--machine` output per language.

## Where it lives

```
ui-tauri/src/i18n/
  config.ts             # supported languages + locale helper (dependency-free)
  resources.ts          # bundles the namespace JSON; lists namespaces
  index.ts              # i18next init (side-effect on import) → default export
  languageBridge.ts     # one-way sync: UI store `lang` → i18next + <html lang>
  react-i18next.d.ts    # type-safe key augmentation (typed off English)
  i18n.test.ts          # en/de key-parity guard
  locales/<lang>/<namespace>.json
ui-tauri/vitest.setup.ts # `import "@/i18n"` so tests resolve real strings
```

The UI store's `lang` (`@/store/ui`, persisted under `kb.ui`) is the **single
source of truth** for the active language. `installLanguageBridge()` (called
once in `main.tsx`) applies it and keeps i18next in sync on every `setLang`.
There is deliberately no browser language detector — one source of truth, no
disagreement. First-run defaults to English (`DEFAULT_LANGUAGE`).

## Using translations in a component

```tsx
import { useTranslation } from "react-i18next";

function Panel() {
  const { t } = useTranslation("settings"); // default namespace for this t()
  return (
    <>
      <h3>{t("appearance.theme.title")}</h3>
      <button>{t("common:actions.reset")}</button>           {/* other ns: prefix */}
      <span>{t("appearance.scale.value", { percent: 90 })}</span> {/* interpolation */}
    </>
  );
}
```

Keys are **type-checked** against the English bundles (`react-i18next.d.ts`), so
a typo or missing key is a compile error and editors autocomplete keys.

## Adding a string

1. Pick a namespace (`common`, `nav`, `settings`, …) or add a new one.
2. Add the key to **both** `locales/en/<ns>.json` and `locales/de/<ns>.json`.
   The bundles must stay in lockstep — `i18n.test.ts` fails otherwise.
3. Reference it with `t("key")` (or `t("ns:key")` from another namespace).

Use interpolation, not string concatenation, so word order can differ per
language: `t("nav:openBook.body", { book, page })` → `"{{book}} is now active.
Opening {{page}}."` / `"{{book}} ist jetzt aktiv. {{page}} wird geöffnet."`.
For counts use i18next plurals (`key_one` / `key_other`).

**German register: `du` (informal).** When copy addresses the user directly,
use the informal imperative — `"Wähle die Sprache…"`, not `"Wählen Sie…"`.
Impersonal/infinitive phrasing is fine where it reads naturally.

**Stable ids vs. display labels.** When a code maps to a label (nav routes,
enum→label maps), keep a stable `id`/slug for test ids and lookups and a
separate `labelKey` for display — never derive a test id or a lookup from a
translated string. See `BOOK_ROUTES` in `routes/BirdsEye.tsx`.

## Keeping English & German in sync (dev workflow)

English and German stay aligned because the tooling makes divergence a
build/test failure, not a silent gap. Internalize this:

**Golden rule — never touch one language alone.** Whenever you add, rename, or
remove a UI string, edit **both** `locales/en/<ns>.json` and
`locales/de/<ns>.json` in the same change: same key path, same structure, same
interpolation placeholders.

**Three guards catch mistakes before they ship:**

1. **Type-safe keys (compile time).** `react-i18next.d.ts` augments **i18next**'s
   `CustomTypeOptions` off the English bundle, so a typo'd or unknown key —
   `t("settings:appearnce.title")` — is a `pnpm typecheck` error, not a silent
   raw-key render. Consequences for how you call `t`:
   - Dynamic keys (`t(map[key])`) need the map typed as literal keys
     (`= {…} as const satisfies Record<K, string>`), not `Record<K, string>`.
   - Genuinely runtime keys (`` t(`range.${x}`) ``) use a narrow `as never` cast
     with a `// dynamic key` comment.
   - Cross-namespace keys need that namespace in the hook:
     `useTranslation(["settings", "common"])` to use `t("common:…")`.
2. **Key-parity test (test time).** `i18n.test.ts` fails if any namespace's `en`
   and `de` key sets differ — a key added to `en` but forgotten in `de` (or
   vice-versa) breaks `pnpm test`. This is what guarantees no half-translated
   namespace.
3. **No silent fallback at runtime.** A missing key renders as the key string
   (`settings:foo.bar`) — i18next's default — and the instance sets
   `returnNull: false` so an explicitly null-valued key also renders its key
   string instead of `null`. Either way a gap is loud in the preview/review,
   never a blank label.

**The loop** (run from `ui-tauri/`):

```bash
pnpm typecheck      # typed keys: typos, missing keys, wrong namespace, bad dynamic keys
pnpm test --run     # en/de key parity + component render tests
pnpm lint           # style
```

CI enforces both under the `verify` check (ci.yml): a `Desktop UI typecheck`
step runs `pnpm typecheck`, and `./scripts/quality-gate.sh` runs the Vitest
suite (including the en/de parity guard). So a typo'd key or a half-translated
namespace cannot merge. (`quality-gate.sh` runs the UI Vitest too, so running
the repo gate locally also exercises the parity check.)

**Terminology — use the glossary, don't improvise.** German is Austrian,
informal `du`. Before inventing a term, check [i18n-glossary.md](i18n-glossary.md):
Bitcoin/space jargon stays English (Wallet, Node, Blockchain, UTXO, Swap,
Peg-in/out, on-chain, Sync, Backend, Provider, Setup, **AI** — not "KI"),
ordinary words translate (Transaktion, Gebühr, Adresse, Guthaben), and the
book/books unit is German (Buch/Bücher). When you settle a *new* term, add it to
the glossary in the same change so it stays consistent across the bundles.

**Review both languages.** Flip the UI with the Settings/header switcher, or in
the browser preview (`pnpm dev:browser`, the mock-daemon mode best suited to
pure language review; `pnpm dev:bridge` talks to the real daemon) seed the
persisted language in devtools
and reload:

```js
const s = JSON.parse(localStorage.getItem("kb.ui") ?? "{}");
s.state = { ...(s.state ?? {}), lang: "de" };
localStorage.setItem("kb.ui", JSON.stringify(s));
location.reload();
```

Number/date formatting follows the active language via `localeForLanguage(lang)`
(de → `de-AT`: decimal comma, `Jänner`, space before `%`); money stays
fiat-driven (see the formatting note above), so the two are intentionally
decoupled.

## Adding a language

1. Add an entry to `SUPPORTED_LANGUAGES` in `config.ts` (`code`, native
   `label`, BCP-47 `locale`). The store type and switcher follow automatically.
2. Create `locales/<code>/<ns>.json` for every namespace (copy English, then
   translate). `i18n.test.ts` tells you exactly which keys are missing.
3. Register the bundles in `resources.ts`.

## Formatting: two locales, on purpose

- **Money** stays *currency-driven* in `@/lib/currency` (`FIAT_LOCALES`): EUR is
  always grouped de-AT style (`€ 1.234,56`) regardless of UI language, because
  that follows the currency convention, not the reader's language.
- **Dates / language-sensitive numbers** follow the *UI language*. Use
  `localeForLanguage(lang)` from `config.ts` for the BCP-47 tag passed to `Intl`
  / `toLocaleString`. (Many call sites still hardcode `en-US` — migrating them
  to `localeForLanguage` is part of the sweep below.)

These are intentionally separate; do not collapse them.

## Scaling path

Resources are **bundled** today (two languages, few namespaces) so switching is
synchronous with no flash. When the payload grows, swap `resources.ts` for
`i18next-http-backend` or dynamic `import()` per namespace and lazy-load — only
`resources.ts` / `index.ts` change; every `t(...)` call stays the same.

## Dependencies & version pins

`i18next` and `react-i18next` are pinned to **exact** versions (not caret
ranges), and `package.json` carries a `pnpm.overrides` entry pinning the
transitive `@babel/runtime`. This is deliberate: `ui-tauri/.npmrc` enforces a
90-day `minimum-release-age`, so all three are pinned to the newest releases
that clear that window (i18next 25.8.18, react-i18next 16.5.8, @babel/runtime
7.29.2). When bumping them, pick a version that is again ≥90 days old; do not
relax the `.npmrc` policy to install something newer without owner approval.

Migrate a whole surface at a time so a screen is never half-translated, and add
